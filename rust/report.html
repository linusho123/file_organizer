The experiment

Feature files are meant to be a language-neutral contract. This tests that literally: the same .feature files that specify the Python tool are used to build a Rust one, and the two are checked for byte-identical behavior. Then the Rust core — deliberately free of any filesystem, clock, or randomness — is compiled five ways.

one pure core → four compiler endpoints
file-organizer-core→ cargo native⇒rust-native (ELF)
file-organizer-core→ wasm32-wasip1⇒wasmtime (JIT)
file-organizer-core→ wasm32-unknown → wasm2c → C⇒ clang (ELF)cosmocc (universal APE)

The wasm2c path translates the WebAssembly back to C and lets a native compiler optimize it; a ~60-line C harness owns stdin/stdout across linear memory, so the wasm module needs no syscalls at all. The cosmocc build is one Actually-Portable Executable carrying both x86-64 and aarch64.

What it says about the workflow

The feature files really were a portable contract.

Nine .feature files drove a Python implementation and an independent Rust one to byte-identical behavior — N-version verification, realized end to end.

A strict runner earns its keep.

gherkin-cargo-test rejected collisions.feature on line 2 — its prose began "When a file being moved…", which the grammar reads as a step. pytest-bdd never noticed. Its "unbound step ⇒ fail" ratchet meant reaching green was the equivalence proof.

Writing the contract down exposed undocumented behavior.

Five real behaviors (trailing-dot, symlink & fifo skips, --version, corrupt-manifest undo) had no scenario. Set B pins them down — the port's contract is now stronger than the suite it came from.

Endpoint independence is real for a pure core.

Factor the logic free of I/O and one source compiles to native, WASI, and wasm2c-via-C with identical output. The endpoint becomes a deployment choice — raw speed, a sandbox, or a single universal binary — not a correctness variable.
